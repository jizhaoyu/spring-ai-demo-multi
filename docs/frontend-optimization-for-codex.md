# 前端优化执行说明

## 目标

对当前项目的静态前端进行一轮小步、可回滚的优化，优先解决以下问题：

1. 无障碍语义不完整
2. 移动端抽屉交互不完整
3. 高频区域整块重绘带来的性能和状态维护风险
4. 首屏信息密度偏高
5. 焦点样式和状态反馈不一致

本轮改动应保持现有产品结构、视觉语言和主链路不变，不做大改版，不引入框架，不重写为组件化方案。

## 仓库范围

仅关注以下文件：

- `src/main/resources/static/index.html`
- `src/main/resources/static/styles.css`
- `src/main/resources/static/app.js`

不要修改后端接口、知识库目录、Spring 编排链路和 fallback 逻辑。

## 执行原则

1. 保持 diff 小
2. 优先补语义和交互，不做视觉大改
3. 不新增无必要抽象
4. 不引入第三方前端库
5. 不改变现有接口协议
6. 不破坏移动端和桌面端现有布局
7. 改完后做最小可用验证

## 优先级

### P0

1. 补全详情切换 Tabs 的无障碍语义
2. 补全文档树的树控件语义
3. 补全移动端文档抽屉的模态交互和焦点管理

### P1

1. 降低高频区域 `innerHTML` 全量重绘的范围
2. 优化聊天区滚动路径上的高频 DOM 查询
3. 收敛 `aria-live` 使用范围，避免整块聊天区被频繁播报

### P2

1. 降低首屏默认信息负担
2. 统一 focus/hover/active 状态样式
3. 优化移动端高度单位，减少 `100vh` 抖动

## 具体任务

## 1. 修复详情切换 Tabs 语义

### 现状

- `detailTabs` 只是普通按钮容器
- 只有 `aria-selected`
- 缺少 `tablist/tab/tabpanel` 语义
- 缺少键盘左右切换

### 目标

把“证据 / 轨迹”切换改成语义完整的 tabs。

### 建议改动

在 `index.html` 中：

- 给 `#detailTabs` 增加 `role="tablist"`
- 给 `#detailTabSources` 和 `#detailTabTrace` 增加 `role="tab"`
- 为两个 tab 分别增加唯一 `id`
- 为对应面板增加 `role="tabpanel"`
- 用 `aria-controls` 和 `aria-labelledby` 建立关联

在 `app.js` 中：

- 在 `renderDetailTabs()` 中同步维护：
- `aria-selected`
- `tabindex`
- `hidden`
- 增加 tabs 的键盘支持：
- `ArrowLeft`
- `ArrowRight`
- `Home`
- `End`

### 验收标准

- 键盘可在两个 tab 间切换
- 读屏器能识别 tablist 和当前选中项
- 切换后对应面板状态正确

## 2. 修复文档树语义

### 现状

- 文档树视觉上是树
- 语义上仍是普通按钮列表
- 文件夹缺少 `aria-expanded` 和 `aria-controls`

### 目标

让知识库树成为可理解、可导航的树结构。

### 建议改动

在 `renderCatalogTree()` 和 `renderTreeNode()` 中：

- 根容器增加 `role="tree"`
- 文件夹节点按钮增加：
- `role="treeitem"`
- `aria-expanded`
- `aria-controls`
- 子节点容器增加 `role="group"`
- 文档节点按钮增加：
- `role="treeitem"`
- `aria-selected` 或清晰的当前选中文本状态
- 考虑增加层级相关语义：
- `aria-level`

如果改动成本可控，再补基础键盘交互：

- `ArrowRight` 展开文件夹
- `ArrowLeft` 收起文件夹
- `Enter/Space` 选中文档或切换文件夹

### 验收标准

- 文件夹展开状态有语义标记
- 树层级关系能被读屏识别
- 至少支持基础键盘展开/收起和选择

## 3. 修复移动端文档抽屉

### 现状

- 抽屉只有视觉打开/关闭
- 缺少模态层语义
- 缺少焦点陷阱
- 关闭后焦点不归位
- 可能发生背景可聚焦

### 目标

把移动端文档抽屉补成可用的轻量模态交互。

### 建议改动

在 `index.html` 中：

- 为抽屉内容容器补充适合的 dialog 语义
- 明确抽屉标题和可访问名称

在 `app.js` 中：

- `toggleDocDrawer()` / `renderDocDrawer()` 补：
- 打开时记录触发按钮
- 打开后焦点落到搜索框或抽屉首个可交互元素
- 关闭后焦点返回 `docDrawerToggle`
- 打开时限制背景区可聚焦
- `Escape` 能稳定关闭抽屉
- 需要时加入最小焦点循环逻辑

在 `styles.css` 中：

- 保持现有布局
- 不要引入复杂动画
- 避免滚动穿透

### 验收标准

- 打开抽屉后焦点进入抽屉
- 关闭后焦点返回触发按钮
- 键盘不会跑到背景主内容
- `Escape` 可关闭

## 4. 收敛高频全量重绘

### 现状

以下区域频繁使用整块 `innerHTML` 重绘：

- 文档树
- 阶段条
- 证据区
- 轨迹区
- 验证摘要

### 目标

先做小步优化，优先减少高频路径的无谓重建，不做大规模重构。

### 建议策略

优先处理以下两类：

1. 高频状态但结构相对稳定的区域
- `stageStrip`
- `validationSummary`

2. 高频滚动或交互区域
- `sourceList`
- `traceList`

### 实施建议

- 保持现有函数名和结构
- 不要一次性把全部渲染逻辑推翻
- 优先改成：
- 结构首次创建
- 后续只更新文本、class、属性
- 如果文档树改动成本过高，可先不做深度性能重构，只补语义

### 验收标准

- 不影响现有功能
- 高频更新区域减少整块重建
- 焦点和滚动状态更稳定

## 5. 优化聊天区滚动高频查询

### 现状

`updateJumpLatestButton()` 每次都会查询 `.message-row` 数量。

### 目标

减少滚动过程中的 DOM 全树查询。

### 建议改动

- 使用状态计数或更轻量的缓存方式
- 在追加消息时更新消息数
- 滚动时只计算是否接近底部，不重复扫描所有消息节点

### 验收标准

- 功能与现在一致
- 滚动中无额外的整树查询依赖

## 6. 收敛 aria-live 范围

### 现状

`chatLog` 整个区域使用 `aria-live="polite"`，流式消息可能导致读屏播报噪音过大。

### 目标

把 live region 缩到真正需要播报的内容。

### 建议改动

- 保留运行状态、提问状态等局部 `aria-live`
- 对聊天记录区，避免整个容器作为 live region
- 可以只让新增系统提示或最终回答节点具备适度播报能力

### 验收标准

- 状态变化仍有必要播报
- 不会因为整块聊天区刷新而造成过度朗读

## 7. 统一焦点样式

### 现状

焦点样式已存在，但分布零散，不同控件一致性一般。

### 目标

统一交互控件的 `focus-visible` 体验。

### 建议改动

在 `styles.css` 中：

- 收敛成一套统一 `focus-visible` 规则
- 覆盖至少以下元素：
- `.toolbar-button`
- `.icon-button`
- `.inline-action`
- `.detail-tab`
- `.workset-chip`
- `.workset-pin`
- `.doc-item-button`
- `.tree-node-toggle`
- `.citation-chip`
- `.prompt-chip`
- `input`
- `textarea`

### 验收标准

- 所有主要交互控件都有清晰且一致的键盘焦点样式
- 不破坏现有 hover 视觉

## 8. 移动端高度单位优化

### 现状

存在 `100vh` 和 `calc(100vh - 80px)`，在移动浏览器地址栏伸缩时可能抖动。

### 目标

增强移动端高度稳定性。

### 建议改动

在 `styles.css` 中：

- 对移动端抽屉等关键区域改用更稳妥的视口单位策略
- 可考虑 `100dvh` 配合兼容兜底
- 避免影响桌面端布局

### 验收标准

- 移动端抽屉在地址栏展开/收起时更稳定
- 无明显布局跳动

## 非目标

本轮不要做这些事：

1. 不重写为 React/Vue
2. 不替换现有视觉主题
3. 不重构整个 `app.js` 为模块化架构
4. 不新增状态管理库
5. 不修改后端接口
6. 不调整知识库数据结构
7. 不做与本任务无关的样式整理

## 建议执行顺序

1. 先做 Tabs 语义
2. 再做文档树语义
3. 再做移动端抽屉焦点管理
4. 再做 `updateJumpLatestButton()` 的轻量优化
5. 再收敛 `aria-live`
6. 最后做 focus 样式统一和移动端高度优化

## 验证要求

至少完成以下验证：

1. 桌面端页面可正常加载
2. 移动端宽度下页面可正常加载
3. 文档抽屉可打开关闭
4. 键盘可切换“证据 / 轨迹”
5. 键盘可操作文档树的基础展开/选择
6. 提问后消息流、证据区、轨迹区不回归
7. `mvnw.cmd test` 通过
8. 如可运行，补一次最小页面联调验证
