# AI 知识库问答控制台 UI/UX 实施方案

## 1. 文档目的

本文档用于指导 `AI 知识库问答控制台` 的界面布局与交互优化实施，目标是在不改变现有 Spring AI 后端主链路的前提下，提升业务用户完成查询任务的速度、可信度判断效率和多场景适配能力。

本文档覆盖：

- 当前界面问题归因
- 分阶段实施方案
- 每项方案的复杂度、ROI、A/B 测试建议
- 前后端改动边界
- 验收标准与埋点建议

## 2. 业务目标

### 2.1 核心目标

- 提升信息获取效率，减少完成查询任务所需步骤和时间
- 增强系统透明度，让 AI 决策过程更可理解、可验证
- 支持从简单查询到复杂分析的不同工作流
- 降低使用门槛，让新手快速上手、专家高效工作

### 2.2 成功指标

- 新手用户首次成功查询时间 `<= 5 分钟`
- 熟练用户平均查询时间减少 `>= 20%`
- 布局相关用户投诉减少 `>= 50%`
- 用户满意度评分提升 `>= 1.5 分`
- 支持团队培训成本降低 `>= 30%`

## 3. 当前界面现状摘要

### 3.1 已有能力

当前控制台已经具备较好的基础能力：

- 三栏结构：文档树、问答区、证据/轨迹区
- 运行状态展示
- 多阶段问答状态条
- 示例问题
- 草稿自动暂存
- 专注模式
- 证据/轨迹切换
- 回答复制与追问建议

### 3.2 主要问题

#### 问题 A：首屏信息竞争严重

当前首屏同时展示：

- 顶部状态卡
- Quick Start 引导
- 阶段条
- 示例问题
- 聊天区
- 右侧空状态面板

结果是“第一步先提问”没有成为最显著动作，首问启动成本偏高。

#### 问题 B：文档树仍是静态导航思路

文档树具备筛选能力，但缺少：

- 最近使用
- 常用文档
- 固定文档
- 任务相关文档推荐

对业务用户来说，目录树过于“结构导向”，而不是“任务导向”。

#### 问题 C：证据与审计分离，验证路径较长

用户在答案、证据和轨迹之间切换时仍需要多次视线跳转和点击，尤其是右侧面板在空状态和窄屏下的价值密度不高。

#### 问题 D：透明度信息还不够“可执行”

当前能看到阶段和置信度，但还不能快速回答：

- 扫描了多少文档
- 为什么低置信度
- 是否走了降级链路
- 下一步应该如何改问

#### 问题 E：缺少专家用户效率层

当前只有基础快捷能力，缺少：

- 全局快捷键面板
- 证据高级复制
- 上下文菜单
- 布局预设
- 工作区快照
- 拖拽分栏

#### 问题 F：移动端优先级仍不够理想

在手机和平板场景中，文档树仍然排在问答区之前，导致首次提问入口被推迟。

## 4. 实施原则

- 效率优先：优先优化首问路径、验证路径和高频重复操作
- 渐进披露：复杂能力只在需要时出现
- 信任增强：低置信度、降级链路和证据不足必须给出明确反馈
- 兼容现有技术栈：以 `static/index.html`、`static/styles.css`、`static/app.js` 为主要改造对象
- 可扩展：为后续国际化、多角色布局和工作区快照预留结构
- 可访问：满足 WCAG 2.1 AA
- 性能可控：不显著增加首屏负担，保持首屏加载时间目标 `< 2 秒`

## 5. 分阶段实施路线

### Phase 1：高 ROI 快速优化

目标：缩短首次提问路径、降低首屏认知负担、提升高频操作效率。

包含方案：

- 方案 1：查询启动区重排
- 方案 2：文档树升级为工作集导航
- 方案 3：键盘与复制效率层
- 方案 4：无障碍、国际化、性能底座补强

### Phase 2：透明度与验证体验升级

目标：提升答案可验证性与系统透明度。

包含方案：

- 方案 5：证据与轨迹合并为验证工作台
- 方案 6：AI 过程解释增强

### Phase 3：专家模式与智能工作区

目标：支持多角色、多任务和高级工作流。

包含方案：

- 方案 7：布局预设与工作区快照
- 方案 8：移动端与跨设备协同深化

## 6. 详细实施方案

### 方案 1：查询启动区重排

#### 问题描述

输入框虽然已经在主区域内，但前面仍被引导、阶段条和示例问题分流，导致“先问一句”不够突出。

#### 解决方案

- 将输入框升级为首屏主操作区，形成粘性 `Ask Bar`
- 将 `Quick Start` 改成情境化提示，仅在以下场景展示：
  - 首次进入
  - 空会话
  - 连续低置信度回答
  - 用户停留但未输入
- 将顶部 `runtime banner` 与阶段信息压缩为一条紧凑状态条
- 首次进入时默认强调：
  - 问题输入
  - 当前文档范围
  - 快捷发送提示

#### 前端实施点

- 调整页面首屏结构顺序
- 为输入区增加粘性定位与紧凑状态摘要
- 将引导卡改为按状态显示而非固定常驻
- 将示例问题改为次级入口

#### 实现复杂度评估

中

#### 预期 ROI

- 新手首次提问时间预计下降 `25% - 35%`
- 首屏滚动深度降低
- 首问放弃率降低

#### A/B 测试建议

- A 版：保留当前首屏结构
- B 版：输入区优先 + 引导按需出现
- 关注指标：
  - `time_to_first_submit`
  - `first_query_success_rate`
  - `first_screen_scroll_depth`

#### 验收标准

- 用户进入页面后无需滚动即可完成首次输入和发送
- 首屏仅保留一个主动作按钮
- 引导不应阻塞输入

### 方案 2：文档树升级为工作集导航

#### 问题描述

当前文档树偏向“知识目录浏览”，但业务用户更常见的工作方式是围绕具体任务快速限定文档范围。

#### 解决方案

- 在文档树上方增加 3 个工作集：
  - 最近使用
  - 常用文档
  - 已固定文档
- 默认展开命中路径和最近访问分支
- 搜索结果高亮关键词
- 支持“一键限定为当前问答范围”
- 移动端将文档树改为抽屉或底部面板，而不是首屏常驻

#### 前端实施点

- 在目录树数据渲染前增加工作集聚合层
- 本地保存最近使用文档和固定文档
- 为命中词增加高亮标记
- 小屏下将左栏切换为抽屉式打开

#### 后端影响

无强制依赖，可先纯前端实现。

#### 实现复杂度评估

中

#### 预期 ROI

- 文档范围限定时间下降 `35% - 45%`
- 零证据回答率有机会下降

#### A/B 测试建议

- A 版：当前目录树
- B 版：工作集优先目录树
- 关注指标：
  - `doc_selection_time`
  - `doc_filter_usage_rate`
  - `queries_with_selected_doc_ratio`

#### 验收标准

- 用户能在 2 次点击内将高频文档加入工作集
- 搜索结果可见高亮
- 小屏文档树不再阻塞首问入口

### 方案 3：键盘与复制效率层

#### 问题描述

当前只有 `Ctrl / Cmd + Enter` 发送和“复制答案”，不够支持高频业务操作。

#### 解决方案

- 增加快捷键帮助面板 `?`
- 增加快捷键：
  - `Ctrl / Cmd + K`：搜索文档
  - `Alt + 1`：切换问答
  - `Alt + 2`：切换证据
  - `Alt + 3`：切换轨迹
  - `Ctrl / Cmd + Shift + C`：复制当前证据
- 为证据片段增加 3 种复制方式：
  - 纯文本
  - 带来源
  - Markdown
- 为回答和证据卡增加上下文菜单

#### 实现复杂度评估

中

#### 预期 ROI

- 熟练用户重复查询效率提升 `10% - 15%`
- 支持团队复制答案与证据的时间显著下降

#### A/B 测试建议

- A 版：现有复制与基础键盘
- B 版：快捷键与高级复制
- 关注指标：
  - `shortcut_usage_rate`
  - `copy_action_success_rate`
  - `time_to_copy_evidence`

#### 验收标准

- 所有核心区域均支持键盘访问
- 快捷键不与浏览器常用快捷键冲突
- 复制输出符合审计和协作需要

### 方案 4：无障碍、国际化、性能底座补强

#### 问题描述

当前具备基础焦点态和 reduced motion 支持，但仍缺少：

- Skip Link
- 文案字典化
- 本地图标资源策略
- 更细的首屏性能优化

#### 解决方案

- 增加 `Skip to main content`
- 把静态文案提取为字典对象，准备中英文切换能力
- 优先将图标字体替换为 SVG 图标方案
- 字体改为本地托管或企业内网兼容方案
- 右栏延迟渲染空状态内容，优先保证主问答首屏

#### 实现复杂度评估

中

#### 预期 ROI

- 满足 WCAG 2.1 AA 的基础要求
- 为多语言扩展减少后续返工
- 降低外部字体和图标资源导致的首屏不稳定风险

#### A/B 测试建议

- A 版：当前资源方式
- B 版：本地化资源 + 主链路优先加载
- 关注指标：
  - `FCP`
  - `LCP`
  - `keyboard_completion_rate`

#### 验收标准

- 键盘用户可跳过顶部导航直达主内容
- 主文案不再散落在 HTML 和 JS 各处
- 首屏加载在常规企业网络下更稳定

### 方案 5：证据与轨迹合并为验证工作台

#### 问题描述

当前右栏将证据和轨迹分开，用户验证答案需要来回切换，视线成本较高。

#### 解决方案

- 将右栏整合为统一的 `验证工作台`
- 默认展示当前答案的验证摘要：
  - 证据条数
  - 置信度
  - 是否降级
  - 关键阶段提示
- 点击引用时打开定位证据抽屉
- 轨迹按阶段展开，支持查看详细决策依据

#### 前端实施点

- 重构右栏结构
- 为回答引用和右栏联动增加更强状态同步
- 空状态从“常驻空卡”改为“按需出现”

#### 后端建议

扩展响应内容以支撑更细粒度证据展示。

#### 实现复杂度评估

高

#### 预期 ROI

- 用户验证一条答案的时间预计下降 `30% - 40%`
- 证据查看和轨迹查看的路径更短

#### A/B 测试建议

- A 版：当前右栏
- B 版：统一验证工作台
- 关注指标：
  - `citation_click_to_validation_time`
  - `trace_open_rate`
  - `answer_acceptance_rate`

#### 验收标准

- 用户从答案点击引用到看到对应证据不超过 1 次额外切换
- 右栏信息结构统一，避免重复空状态

### 方案 6：AI 过程解释增强

#### 问题描述

当前透明度信息以“阶段完成”和“置信度”展示为主，但还不足以帮助业务用户判断下一步动作。

#### 解决方案

- 在阶段条或验证工作台中增加：
  - 扫描文档数
  - 命中文档数
  - 降级原因
  - 置信度原因标签
  - 推荐下一步动作
- 对低置信度回答增加显式修正路径：
  - 更换关键词
  - 缩小范围
  - 选择文档
  - 继续追问

#### 后端数据扩展建议

建议在响应模型中增加以下字段：

- `documentsScanned`
- `matchedDocuments`
- `degradedReason`
- `confidenceReason`
- `selectedStrategy`

#### 实现复杂度评估

中到高

#### 预期 ROI

- 降低“为什么不准”的解释成本
- 低置信度重试成功率提升
- 降低培训和投诉成本

#### A/B 测试建议

- A 版：当前置信度展示
- B 版：增加解释性标签与建议动作
- 关注指标：
  - `low_confidence_retry_rate`
  - `retry_success_rate`
  - `trust_feedback_score`

#### 验收标准

- 每个低置信度回答都应给出明确下一步建议
- 降级链路必须显式展示

### 方案 7：布局预设与工作区快照

#### 问题描述

当前只有专注模式，没有角色布局、工作区记忆和任务切换能力，不利于专家用户持续高效使用。

#### 解决方案

- 提供布局预设：
  - 研究员模式
  - 审核员模式
  - 支持人员模式
- 提供工作区快照：
  - 面板开关状态
  - 当前 detail view
  - 当前文档范围
  - 输入草稿
- 支持拖拽调整栏宽
- 支持保存用户个人布局偏好

#### 实现复杂度评估

高

#### 预期 ROI

- 重度用户整体查询时间有机会达到 `>= 20%` 的下降目标
- 多任务切换成本明显下降

#### A/B 测试建议

- A 版：统一布局
- B 版：布局预设 + 工作区快照
- 关注指标：
  - `layout_switch_usage_rate`
  - `session_task_count`
  - `repeat_user_efficiency_gain`

#### 验收标准

- 用户可在 1 次操作内切换到合适布局
- 刷新页面后可恢复上次工作区

### 方案 8：移动端与跨设备协同深化

#### 问题描述

当前移动端虽然有收敛，但仍然以桌面布局压缩为主，而不是重新定义移动优先的任务顺序。

#### 解决方案

- 移动端首屏顺序改为：
  - 状态摘要
  - 输入区
  - 阶段摘要
  - 验证工作台
  - 文档树抽屉
- 保留横向滚动示例问题，不完全隐藏
- 平板支持左右切换和更大点击目标
- 后续支持多设备会话同步和草稿同步

#### 实现复杂度评估

中到高

#### 预期 ROI

- 移动端首次提问时间显著下降
- 移动端完成率提升

#### A/B 测试建议

- A 版：当前移动端排布
- B 版：输入优先的移动端布局
- 关注指标：
  - `mobile_time_to_first_submit`
  - `mobile_query_completion_rate`
  - `mobile_scroll_before_input`

#### 验收标准

- 手机端无需先浏览完整文档树即可直接提问
- 所有主要操作目标符合最小点击面积要求

## 7. 响应契约扩展建议

为支持更强的验证体验，建议扩展当前证据与过程数据结构。

### 7.1 Citation 扩展建议

当前字段：

- `sourceId`
- `title`
- `excerpt`

建议新增：

- `path`
- `heading`
- `score`
- `chunkIndex`
- `documentId`
- `documentLabel`

### 7.2 AgentResponse / Audit 扩展建议

建议补充：

- `documentsScanned`
- `matchedDocuments`
- `degradedReason`
- `confidenceReason`
- `selectedStrategy`
- `estimatedLatency`

## 8. 埋点建议

建议新增以下事件：

- `query_started`
- `query_submitted`
- `query_completed`
- `query_failed`
- `doc_selected`
- `doc_pinned`
- `citation_opened`
- `trace_opened`
- `low_confidence_action_clicked`
- `shortcut_used`
- `layout_preset_switched`
- `workspace_snapshot_loaded`

建议统计属性：

- `viewport_mode`
- `focus_mode`
- `detail_view`
- `has_selected_doc`
- `confidence_level`
- `runtime_status`
- `degraded`

## 9. 推荐实施顺序

### 第一优先级

- 查询启动区重排
- 文档树工作集导航
- 键盘与复制效率层
- 无障碍、国际化、性能底座补强

### 第二优先级

- 验证工作台
- AI 过程解释增强

### 第三优先级

- 布局预设与工作区快照
- 移动端与跨设备协同深化

## 10. 交付清单建议

### 设计交付物

- 页面信息架构图
- 桌面/平板/手机三端线框图
- 状态系统与置信度标识规范
- 快捷键与上下文菜单规范
- 布局预设说明

### 前端交付物

- `index.html` 结构调整
- `styles.css` 布局与响应式优化
- `app.js` 状态、交互和持久化增强
- 本地图标/字体资源优化

### 后端交付物

- `Citation` 扩展
- `AgentResponse` / `AuditLog` 扩展
- 更丰富的透明度字段输出

## 11. 风险与注意事项

- 不要为了提升透明度而让页面再次信息过载
- 不要让高级能力干扰基础提问链路
- 所有新增状态说明都要有明确用户价值，避免“技术自嗨”
- 移动端优化必须重新定义顺序，而不是简单缩放桌面布局
- 新增数据字段时要兼容 fallback 链路，不能让本地降级模式失效

## 12. 结论

本次优化的核心不是“增加更多面板”，而是将现有能力重新组织成更符合业务用户心智的任务流：

- 先快速提问
- 再快速验证
- 最后按需深挖

建议先执行 Phase 1，在不引入高风险改造的前提下尽快兑现效率提升；待主链路稳定后，再推进验证工作台和专家级工作区能力。
